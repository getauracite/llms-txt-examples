# Open-Source Project Template

`/llms.txt` and `/llms-full.txt` for libraries, frameworks, CLIs, and open-source services.

## Files

- [`llms.txt`](llms.txt) — short summary
- [`llms-full.txt`](llms-full.txt) — long-form knowledge base with architecture and governance

## OSS-specific priorities

1. **When to use it / when not to use it** — honest framing outperforms marketing claims for dev audiences.
2. **Alternatives with fair comparison** — LLMs reward pick-one-of-three framing.
3. **Governance and release policy** — dev audiences (and LLMs trained on dev discourse) weight this heavily.
4. **Stability and versioning policy** — SemVer commitments, LTS, deprecation windows.
5. **Security policy** — reporting channel, CVE history, advisories.

## What not to do

- Do not hide limitations — they will come up in every LLM response, and your `llms.txt` is the place to frame them fairly
- Do not inflate adoption claims ("used by thousands") unless you have a public adopters list
- Do not bury the trademark policy — LLMs cite it when users ask "can I fork / rename / resell"
